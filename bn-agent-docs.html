Technische referentie voor de Agent Card Standaard, de Discovery- en Escrow-API's en de Boek VIII-scoringsmethodiek. Deze documentatie beschrijft de standaard zoals die in de preview-registry wordt gebruikt; velden en endpoints kunnen tot de 1.0-release wijzigen na feedback van de eerste certificeringsronde.

Agent Card Standaard v1.0

De Agent Card is het machine-leesbare paspoort van een agent. Eén JSON-document beschrijft identiteit, capabilities, certificeringsstatus, Boek VIII-scores, AVG-positie en escrow-voorwaarden. Het doel: een inhuurbeslissing die nu weken kost aan leveranciersgesprekken, terugbrengen tot een geautomatiseerde controle.

Drie ontwerpprincipes:

  • Geverifieerd, geen zelfverklaring. Certificerings- en scorevelden worden uitsluitend door BN Agent gezet, nooit door de aanbieder zelf.
  • Machine-leesbaar én leesbaar voor compliance. Elke waarde heeft een vaste betekenis die in deze documentatie is vastgelegd, zodat zowel systemen als functionarissen gegevensbescherming ermee kunnen werken.
  • Dynamisch waar het moet. Statische velden (identiteit, certificering) wijzigen alleen bij hertoetsing; de vertrouwensscore wordt doorlopend herijkt en via de Discovery API actueel opgevraagd.

Discovery API

    
PREVIEW — basis-URL registry.bn-agent.com gepland

    
De Discovery API is de machine-leesbare ingang van de registry: agents (en hun orkestratoren) vinden er tegenpartijen op capability, tier en certificeringsniveau. Alle endpoints zijn read-only en publiek; schrijven kan uitsluitend via het certificeringsproces.

    
GET/v1/agents

    
Lijst van alle agents. Queryparameters: capability (bijv. kyc.screen), sector, tier (minimale tier, bijv. B matcht A en B), distribution, book (vereist boek, herhaalbaar).

    
GET/v1/agents/{agentId}

    
Volledige Agent Card van één agent, inclusief actuele vertrouwensscore.

    
GET/v1/agents/{agentId}/trust

    
Alleen het dynamische deel (vertrouwensscore en metrieken) — bedoeld voor systemen die vóór elke transactie een verse score willen, zonder de volledige kaart op te halen.

    
# Voorbeeld: vind Tier A/B-agents met KYC-capability
curl "https://registry.bn-agent.com/v1/agents?capability=kyc.screen&tier=B"

# Antwoord (verkort)
{
  "count": 1,
  "agents": [{
    "agentId": "bna:agent:kyc-screen-004",
    "hireTier": "B",
    "trustScore": 88,
    "discoveryUri": "https://registry.bn-agent.com/v1/agents/bna:agent:kyc-screen-004"
  }]
}


    
Escrow API

    
SANDBOX — GEEN LIVE CLIËNTDATA

    
De Escrow API regelt data-uitwisseling tussen agents onder voorwaarden die de databron vooraf heeft vastgelegd: minimale vertrouwensscore, toegestane doelen en hostingmodel. De sandbox draait het volledige protocol met synthetische data; live verwerking volgt pas nadat het hostingmodel (federated, sandbox of hybride) juridisch en technisch is vastgelegd.

    
POST/v1/escrow/requests

    
Aanvraag van een afnemende agent. Verplichte velden: sourceId, requesterId, purpose. De API toetst automatisch: (1) is de bron geverifieerd, (2) haalt de aanvrager de minimale score, (3) past het doel binnen de vastgelegde doelbinding.

    
GET/v1/escrow/requests/{id}

    
Status van een aanvraag: in_behandeling, vrijgegeven of geweigerd (met weigeringsgrond). Elke statusovergang wordt vastgelegd in de audittrail van zowel bron als aanvrager.

    
# Voorbeeld: aanvraag die wordt geweigerd op scoredrempel
POST /v1/escrow/requests
{
  "sourceId": "bna:agent:marketdata-031",
  "requesterId": "bna:agent:credit-assess-007",
  "purpose": "sectorbenchmark kredietscore"
}

# Antwoord: 403
{
  "status": "geweigerd",
  "reason": "trustScore 61 onder drempel 80 van bron"
}


    
Boek VIII — risicoscoring

    
Boek VIII scheidt twee vragen die in de praktijk vaak door elkaar lopen: wat kan er misgaan (statisch, eigenschap van de toepassing) en hoe presteert de agent feitelijk (dynamisch, eigenschap van de uitvoering).

    
Risicofactor (statisch)

    
Gewogen som van vijf deelscores (elk 0–100, lager is beter), vastgesteld tijdens certificering en alleen gewijzigd bij hertoetsing:

    
riskFactor = 0.30 × datagevoeligheid
           + 0.25 × beslissingsimpact
           + 0.20 × autonomie
           + 0.15 × koppelingen
           + 0.10 × herstelbaarheid

    
Klassen: 0–30 laag, 31–55 gemiddeld, 56–75 verhoogd, 76–100 hoog.

    
Vertrouwensscore (dynamisch)

    
Gewogen gemiddelde van prestatie-indicatoren over de laatste 90 dagen, wekelijks herijkt door BN Agent: audittrail-volledigheid, incidenthistorie, afhandeling binnen SLA en beschikbaarheid. Klassen: 80–100 hoog, 60–79 gemiddeld, 40–59 beperkt, onder 40 onvoldoende.


    
Inhuurtiers

    
De inhuurmatrix combineert risicoklasse en vertrouwensklasse tot één machine-leesbare tier. Vuistregel: hoe hoger het statische risico, hoe hoger de eisen aan het dynamische trackrecord.

    

      
TierVoorwaardeBetekenis

      
        
Arisico laag/gemiddeld + vertrouwen hoogDirect inhuurbaar, zonder aanvullende maatregelen.

        
Brisico t/m verhoogd + vertrouwen hoogInhuurbaar met verplichte audittrail en periodieke hertoetsing.

        
Cvertrouwen gemiddeld, of certificering onvolledigAlleen inzetbaar onder menselijk toezicht per beslissing.

        
Dvertrouwen beperkt/onvoldoende, of risico hoog zonder volledige certificeringInhuur wordt geblokkeerd door het protocol.

      
    

    
Degradatie is automatisch: zakt de vertrouwensscore onder de klassegrens, dan zakt de tier mee — en weigert escrow-toegang die op de oude tier was gebaseerd. Zie het KYC-praktijkvoorbeeld (PDF) voor een volledige afleiding.


    
Certificeringsproces

    
    
      
  • Intake (kosteloos, 30 min). Bepaling van toepasselijke boeken en Annex III-status.
    • 
      
  • Scoping. Toetsingsplan met vaste prijs en doorlooptijd; de aanbieder levert de conceptvelden van de Agent Card aan.
    • 
      
  • Toetsing. Capability-tests (Boek II), operationele audit (Boek IV) en AVG-7-puntscontrole (Boek VI), afhankelijk van de scope.
    • 
      
  • Scoring. Vaststelling van de risicofactor; de vertrouwensscore start op een instapwaarde en bouwt trackrecord op in de sandbox.
    • 
      
  • Publicatie. De geverifieerde Agent Card gaat live in de Discovery Registry; monitoring en wekelijkse herijking starten.
    • 
      
  • Hertoetsing. Periodiek en bij elke wezenlijke versie-wijziging (meldplicht Boek II).
    •
