CONCEPT-DOCUMENTATIE — STANDAARD V2.0, API'S IN PREVIEW

BN Agent documentatie

Technische referentie voor de Agent Card Standaard, de Discovery- en Escrow-API's en de Boek VIII-scoringsmethodiek. Deze documentatie beschrijft de standaard zoals die in de preview-registry wordt gebruikt; velden en endpoints kunnen tot de 1.0-release wijzigen na feedback van de eerste certificeringsronde.

Agent Card Standaard v2.0

De Agent Card is het machine-leesbare paspoort van een agent. Eén JSON-document beschrijft identiteit, capabilities, certificeringsstatus, Boek VIII-scores, AVG-positie en escrow-voorwaarden. Het doel: een inhuurbeslissing die nu weken kost aan leveranciersgesprekken, terugbrengen tot een geautomatiseerde controle.

Drie ontwerpprincipes:

  • Geverifieerd, geen zelfverklaring. Certificerings- en scorevelden worden uitsluitend door BN Agent gezet, nooit door de aanbieder zelf.
  • Machine-leesbaar én leesbaar voor compliance. Elke waarde heeft een vaste betekenis die in deze documentatie is vastgelegd, zodat zowel systemen als functionarissen gegevensbescherming ermee kunnen werken.
  • Dynamisch waar het moet. Statische velden (identiteit, certificering) wijzigen alleen bij hertoetsing; de vertrouwensscore wordt doorlopend herijkt en via de Discovery API actueel opgevraagd.

v2.0 voegt één veld toe aan v1.1: connectorIds — verwijzingen naar gecertificeerde systeemconnectors. Alle v1.1-velden zijn ongewijzigd; v1.1-manifesten blijven geldig (manifest_version accepteert 1.1 en 2.0).

Datablokken v1.1 — de vier verplichte blokken

Om te voldoen aan de Wft en de EU AI Act (transparantie over hoog-risico-AI) bevat elke Agent Card sinds v1.1 vier datablokken plus een integriteitsblok en een risico-barometer. De risico-barometer (riskLevel, 1–5) werkt zoals de risico-indicator bij beleggingsfondsen: niveau 1 = alleen interne dataconsumptie, niveau 5 = hoog risico (BSN/UBO, KYC of externe transactie-API's).

Connectors — gecertificeerde systeemkoppelingen

Een connector is een gecertificeerde koppeling naar een extern systeem (iDIN, KVK Handelsregister, DigiD, eHerkenning, Microsoft Graph, Salesforce, …) met een eigen manifest: schemas/connector.schema.json. Agents declareren hun koppelingen via connectorIds op de Agent Card; de connectorcatalogus toont alle voorbeeldconnectors. Twee regels maken de laag verifieerbaar:

  • Risicobijdrage telt mee. Elke connector heeft een riskContribution (0–100). Invariant: riskBreakdown.links van elke gebruikende agent is ten minste de hoogste riskContribution van zijn connectors — een agent kan nooit een lichtere koppelingsscore claimen dan zijn zwaarste koppeling.
  • Elke aanroep in de audittrail. Een aanroep via een connector landt als input_source met connector_id in de WORM-audittrail, inclusief response_hash. De auditor ziet per transactie welke externe systemen zijn geraakt.

Veldreferentie connector-manifest

authTypeenumoauth2 (OAuth 2.1, per-afnemer client credentials), apiKey (sleutel in de kluis van de hostingomgeving) of mtls (wederzijds TLS-certificaat, o.a. DigiD/eHerkenning/BKR).
scopesstring[]Maximale scopes die de connector mag aanvragen; een agent krijgt nooit meer, minder mag altijd.
dataCategoriesstring[]Datacategorieën die via de koppeling kunnen stromen; bepaalt mede de vereiste certificeringsboeken van de agent.

Credentials staan nooit in het manifest of het register: geheimen leven uitsluitend in de kluis van de hostingomgeving (Optie A: platform-kluis per tenant; Optie B: de kluis van de afnemer). Zie de veiligheidsklep CONNECTOR_LIVE_CREDENTIALS bij de Connectors API.

Discovery API

    
PREVIEW — basis-URL registry.bn-agent.com gepland

    
De Discovery API is de machine-leesbare ingang van de registry: agents (en hun orkestratoren) vinden er tegenpartijen op capability, tier en certificeringsniveau. Alle endpoints zijn read-only en publiek; schrijven kan uitsluitend via het certificeringsproces.

    
GET/v1/agents

    
Lijst van alle agents. Queryparameters: capability (bijv. kyc.screen), sector, tier (minimale tier, bijv. B matcht A en B), distribution, book (vereist boek, herhaalbaar).

    
GET/v1/agents/{agentId}

    
Volledige Agent Card van één agent, inclusief actuele vertrouwensscore.

    
GET/v1/agents/{agentId}/trust

    
Alleen het dynamische deel (vertrouwensscore en metrieken) — bedoeld voor systemen die vóór elke transactie een verse score willen, zonder de volledige kaart op te halen.

    
# Voorbeeld: vind Tier A/B-agents met KYC-capability
curl "https://registry.bn-agent.com/v1/agents?capability=kyc.screen&tier=B"

# Antwoord (verkort)
{
  "count": 1,
  "agents": [{
    "agentId": "bna:agent:kyc-screen-004",
    "hireTier": "B",
    "trustScore": 88,
    "discoveryUri": "https://registry.bn-agent.com/v1/agents/bna:agent:kyc-screen-004"
  }]
}


    
Connectors API

    
PREVIEW — CREDENTIAL-BEHEER ACHTER FEATURE FLAG CONNECTOR_LIVE_CREDENTIALS

    
De Connectors API ontsluit de catalogus. Lezen is publiek, net als de registry: een afnemer moet vóór inhuur kunnen beoordelen welke koppelingen een agent gebruikt. Operationele detailvelden (endpointallowlist, certificeringsrapport) zijn alleen zichtbaar voor geauthenticeerde aanroepers.

    
GET/v1/connectors

    
Catalogus doorzoeken. Queryparameters: q, auth_type (oauth2 | apiKey | mtls), category, data_residency, status, limit (1–100), offset.

    
GET/v1/connectors/{connectorId}

    
Volledig manifest + de agents die de connector gebruiken. Veldzichtbaarheid per aanroeper, zoals bij /v1/agents/{agentId}.

    
POST/v1/connectors

    
Manifest indienen/updaten (scope connectors:write). Uitsluitend metadata: payloads met credentials, secrets of apiKey worden categorisch geweigerd (422) zolang CONNECTOR_LIVE_CREDENTIALS uit staat.

    
# Voorbeeld: alle mTLS-connectors met NL-residentie
curl "https://registry.bn-agent.com/v1/connectors?auth_type=mtls&data_residency=NL_only"

# Antwoord (verkort)
{
  "count": 3,
  "connectors": [{
    "connectorId": "bnc:connector:digid",
    "authType": "mtls",
    "riskContribution": 60,
    "status": "actief"
  }]
}


    
Escrow API

    
SANDBOX — GEEN LIVE CLIËNTDATA

    
De Escrow API regelt data-uitwisseling tussen agents onder voorwaarden die de databron vooraf heeft vastgelegd: minimale vertrouwensscore, toegestane doelen en hostingmodel. De sandbox draait het volledige protocol met synthetische data; live verwerking volgt pas nadat het hostingmodel (federated, sandbox of hybride) juridisch en technisch is vastgelegd.

    
POST/v1/escrow/requests

    
Aanvraag van een afnemende agent. Verplichte velden: sourceId, requesterId, purpose. De API toetst automatisch: (1) is de bron geverifieerd, (2) haalt de aanvrager de minimale score, (3) past het doel binnen de vastgelegde doelbinding.

    
GET/v1/escrow/requests/{id}

    
Status van een aanvraag: in_behandeling, vrijgegeven of geweigerd (met weigeringsgrond). Elke statusovergang wordt vastgelegd in de audittrail van zowel bron als aanvrager.

    
# Voorbeeld: aanvraag die wordt geweigerd op scoredrempel
POST /v1/escrow/requests
{
  "sourceId": "bna:agent:marketdata-031",
  "requesterId": "bna:agent:credit-assess-007",
  "purpose": "sectorbenchmark kredietscore"
}

# Antwoord: 403
{
  "status": "geweigerd",
  "reason": "trustScore 61 onder drempel 80 van bron"
}


    
Audit API

    
PREVIEW — WORM-BACKEND ACHTER FEATURE FLAG AUDIT_WORM_BACKEND

    
De Audit API is de logstream van het platform. Agents pushen bij afronding van elke taak een audit-log-entry via een beveiligde HTTPS-stream (gRPC-variant beschikbaar) naar de onwijzigbare WORM-opslag — AWS QLDB of Azure Immutable Blob Storage. Elke entry bevat de hash van de vorige entry, zodat zelfs de beheerder van BN Agent een logboek niet ongemerkt kan wissen of aanpassen. Bekijk het auditor-dashboard met voorbeelddata.

    
POST/v1/audit/logs

    
Entry indienen (scope audit:write). De payload wordt gevalideerd tegen het audit-log-entry-schema. De identities.agent.code_hash moet overeenkomen met de ondertekende Agent Card; bij een mismatch wordt de entry geweigerd én de agent bevroren.

    
GET/v1/audit/logs

    
Audittrail doorzoeken (scope audit:read, alleen eigen organisatie tenzij auditor-mandaat). Queryparameters: agent_id, transaction_id, decision (APPROVED | REJECTED | OVERRIDE | AUTONOOM), from, to, limit, offset.

    
GET/v1/audit/logs/{transaction_id}

    
Eén volledige entry inclusief chainProof: de herberekening van entry_hash en prev_entry_hash waarmee een externe auditor de onwijzigbaarheid onafhankelijk vaststelt.


    
Certificaatcheck

    
PUBLIEK — GEEN AUTHENTICATIE

    
Wanneer een adviseur een transactie-benchmark of financieel model deelt met een klant, accountant of externe auditor, staat onder het document een gecertificeerd watermerk: “Geproduceerd via een BN-compliant agent — controleer de Agent Card en de HITL-audittrail via be-an-agent.nl/check/<id>”. Dat maakt elke output zelf-verifieerbaar — en elke gedeelde output een aanbeveling van het netwerk. Probeer de publieke checkpagina.

    
GET/v1/check/{agentId}

    
Certificerings- en integriteitsstatus van één agent: is de agent gecertificeerd, is de code_hash nog intact (of gebroken en dus bevroren), risiconiveau en tier, en het aantal audittrail-entries. Met ?tx={transaction_id} wordt tevens gecontroleerd of de transactie in het WORM-logboek van deze agent staat.

    
# Voorbeeld: check inclusief transactieverificatie
curl "https://bn-agent.com/v1/check/bna:agent:kyc-screen-004?tx=tx-baa-2026-8839017"

# Antwoord (verkort)
{
  "certification": { "status": "gecertificeerd", "validUntil": "2027-03" },
  "integrity": { "status": "geldig", "codeHashValid": true },
  "riskLevel": 4,
  "hireTier": "B",
  "transaction": { "found": true, "decision": "APPROVED" }
}


    
Logvelden — de vier verplichte categorieën

    
Elke audit-log-entry documenteert vier categorieën, zodat de audittrail waterdicht is voor een interne compliance-auditor én de toezichthouder:

    
    
      
  • Context & identificatie (wie en wanneer). log_header + identities: timestamp (UTC), transactie- en sessie-ID, identiteit/rol/rechten van de menselijke gebruiker, en agent-ID + versie + code_hash gekoppeld aan de Agent Card. Dit is de onweerlegbaarheid: de code_hash bewijst dat de agent exact functioneerde zoals gecertificeerd, de user_role dat de gebruiker bevoegd was.
    • 
      
  • AI-logica & data lineage (de gedachtegang). data_lineage: gepseudonimiseerde invoer, bronnen met response_hash (wijzigt de marktdata later, dan blijft de historische beslissing traceerbaar) en — bij aanroepen via een systeemconnector — het connector_id, modeloutput met confidence-score en de verplichte chain_of_thought die het black-box-effect wegneemt.
    • 
      
  • Human-in-the-loop & overrides (de menselijke factor). human_in_the_loop: gate-registratie, reviewduur en de beslissing APPROVED/REJECTED/OVERRIDE. Bij een override bevriest de interface tot de verplichte motivatie is ingetypt; oude (AI) en nieuwe (mens) waarde worden beide vastgelegd, afgesloten met de digitale handtekening waarmee de adviseur de Wft-eindverantwoordelijkheid aanvaardt.
    • 
      
  • Systeemintegriteit & ICT-weerbaarheid (de DORA-check). system_integrity + compliance_integrity: responstijden, ongewone token-consumptie (indicatie van een aanval), technische fouten mét hersteld-door-veld, AI Act-risicocategorie en toegepaste PII-masking.
    • 
    


    
JSON-schema's

    
De schema's zijn JSON Schema draft 2020-12 en worden in de certificeringspipeline en de Audit API afgedwongen:

    
    
      
  • audit-log-entry.schema.json — de WORM-logpayload (voorbeeld). Conditionele regels: decision: OVERRIDE vereist human_override_details + handtekening; een geactiveerde HITL-gate vereist gate-metadata; source_type: api vereist een connector_id uit de catalogus.
    • 
      
  • agent-manifest.schema.json — het manifest dat de ontwikkelaar met de agentcode indient (voorbeeld). Claims als data_residency en zero_data_retention worden in stap 1 van de certificeringspipeline geautomatiseerd gecontroleerd; high_risk vereist een DPIA. v2.0 voegt het optionele architecture.connector_ids toe.
    • 
      
  • connector.schema.json — het connector-manifest (voorbeeld). Zie de veldreferentie; riskContribution wordt bij certificering van elke gebruikende agent getoetst tegen riskBreakdown.links.
    • 
    


    
Boek VIII — risicoscoring

    
Boek VIII scheidt twee vragen die in de praktijk vaak door elkaar lopen: wat kan er misgaan (statisch, eigenschap van de toepassing) en hoe presteert de agent feitelijk (dynamisch, eigenschap van de uitvoering).

    
Risicofactor (statisch)

    
Gewogen som van vijf deelscores (elk 0–100, lager is beter), vastgesteld tijdens certificering en alleen gewijzigd bij hertoetsing:

    
riskFactor = 0.30 × datagevoeligheid
           + 0.25 × beslissingsimpact
           + 0.20 × autonomie
           + 0.15 × koppelingen
           + 0.10 × herstelbaarheid

    
De koppelingen-deelscore heeft sinds v2 een harde ondergrens: hij is ten minste de hoogste riskContribution van de connectors die de agent via connectorIds declareert.

    
Klassen: 0–30 laag, 31–55 gemiddeld, 56–75 verhoogd, 76–100 hoog.

    
Vertrouwensscore (dynamisch)

    
Gewogen gemiddelde van prestatie-indicatoren over de laatste 90 dagen, wekelijks herijkt door BN Agent: audittrail-volledigheid, incidenthistorie, afhandeling binnen SLA en beschikbaarheid. Klassen: 80–100 hoog, 60–79 gemiddeld, 40–59 beperkt, onder 40 onvoldoende.


    
Inhuurtiers

    
De inhuurmatrix combineert risicoklasse en vertrouwensklasse tot één machine-leesbare tier. Vuistregel: hoe hoger het statische risico, hoe hoger de eisen aan het dynamische trackrecord.

    

      
TierVoorwaardeBetekenis

      
        
Arisico laag/gemiddeld + vertrouwen hoogDirect inhuurbaar, zonder aanvullende maatregelen.

        
Brisico t/m verhoogd + vertrouwen hoogInhuurbaar met verplichte audittrail en periodieke hertoetsing.

        
Cvertrouwen gemiddeld, of certificering onvolledigAlleen inzetbaar onder menselijk toezicht per beslissing.

        
Dvertrouwen beperkt/onvoldoende, of risico hoog zonder volledige certificeringInhuur wordt geblokkeerd door het protocol.

      
    

    
Degradatie is automatisch: zakt de vertrouwensscore onder de klassegrens, dan zakt de tier mee — en weigert escrow-toegang die op de oude tier was gebaseerd. Zie het KYC-praktijkvoorbeeld (PDF) voor een volledige afleiding.


    
Certificeringsproces

    
    
      
  • Intake (kosteloos, 30 min). Bepaling van toepasselijke boeken en Annex III-status.
    • 
      
  • Scoping. Toetsingsplan met vaste prijs en doorlooptijd; de aanbieder levert de conceptvelden van de Agent Card aan.
    • 
      
  • Toetsing. Capability-tests (Boek II), operationele audit (Boek IV) en AVG-7-puntscontrole (Boek VI), afhankelijk van de scope.
    • 
      
  • Scoring. Vaststelling van de risicofactor; de vertrouwensscore start op een instapwaarde en bouwt trackrecord op in de sandbox.
    • 
      
  • Publicatie. De geverifieerde Agent Card gaat live in de Discovery Registry; monitoring en wekelijkse herijking starten.
    • 
      
  • Hertoetsing. Periodiek en bij elke wezenlijke versie-wijziging (meldplicht Boek II).
